Die Gesamtarchitektur: vier Schichten

Die drei Datenflüsse

Architekturdiagramme zeigen, wer existiert. Datenflüsse zeigen, was bei einem Request wirklich passiert. Diese drei Abläufe sind das Herz des Systems — wenn du sie erklären kannst, verstehst du RAG.

Agenten-Pipeline — deine LEGO-Rollen als Code

POST /api/agents/run · AgentPipeline.run() — Vorstufe zu LangGraph (Phase 4)

Rolle 1PlannerZerlegt das Briefing in max. 5 nummerierte Schritte
→
Rolle 2DeveloperSetzt den Plan um — bekommt Briefing + Plan als Input
→
Rolle 3ReviewerPrüft, verbessert, liefert die finale Fassung
Warum sequenziell statt LangGraph?

Erst das Muster verstehen (Output von Agent A = Input von Agent B), dann das Framework lernen. LangGraph fügt später Schleifen („Tester lehnt ab → zurück zum Developer"), Zustand und Human Approval hinzu — aber der Kern bleibt genau diese Kette.

03

Wer kennt wen? Die Abhängigkeits-Regeln

Das ist der Teil, der ein „Skript" von einer „Plattform" unterscheidet. Jede Komponente hat einen klar definierten Platz — und darf nur bestimmte andere kennen.

KomponenteKennt (hängt ab von)Kennt bewusst NICHTDarum
routes.py Modelle, deps.py, Service-Interfaces Ollama, httpx, Dateipfade Routen nur übersetzen: HTTP ↔ Python. Logik lebt in Services.
RagService Parser, Chunker, Store, LLMClient-Protocol FastAPI, Request, HTTP-Statuscodes Dadurch im Test ohne Webserver direkt aufrufbar.
OllamaClient httpx, Ollama-API-Format RAG, Agenten, Routen Einziger Ort, der weiß, wie man mit Ollama spricht. Provider-Wechsel = 1 Datei.
JsonVectorStore ChunkRecord/DocumentMeta, Dateisystem Embeddings-Herkunft, LLM Bekommt fertige Vektoren. Qdrant kann mit identischer Schnittstelle daneben.
main.py (Lifespan) alles — er verdrahtet die Objekte — Die „Composition Root": der einzige Ort, an dem konkrete Klassen zusammengesteckt werden.
Tests App + FakeLLMClient Netzwerk, echtes Ollama Deterministisch & schnell: 13 Tests in 0,4 s, laufen auch offline und in CI.
Das Schlüsselkonzept: Dependency Injection

Kein Service erzeugt seine Abhängigkeiten selbst — er bekommt sie im Konstruktor (RagService(llm=..., store=...)). Beim Serverstart steckt main.py die echten Teile zusammen (OllamaClient, JSON-Store); im Test steckt conftest.py Fakes hinein. Gleicher Code, andere Verkabelung. Das ist die eine Idee, die fast alle anderen Entscheidungen hier erklärt.

Konfiguration als typisierte Settings

app/core/config.py

Alle Stellschrauben (Ollama-URL, Modellnamen, Chunk-Größe) leben in einer Settings-Klasse. Werte kommen aus Umgebungsvariablen mit Prefix AIOS_ oder einer .env-Datei.

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix="AIOS_", env_file=".env")
    ollama_url: str = "http://localhost:11434"
    chat_model: str = "llama3"
    chunk_size: int = 800

@lru_cache  # wird nur einmal gelesen
def get_settings() -> Settings: return Settings()
Warum?

12-Factor-Prinzip: Konfiguration gehört in die Umgebung, nicht in den Code. AIOS_CHAT_MODEL=mistral setzen — fertig, kein Code-Change. Und weil es Pydantic ist, crasht ein Tippfehler beim Start statt irgendwann zur Laufzeit.

Datenmodelle: Validierung an der Grenze

app/models/chat.py · documents.py · rag.py

Jede Ein- und Ausgabe der API ist ein Pydantic-v2-Modell. ChatRequest lehnt leere Nachrichten ab, RagQuery begrenzt top_k auf 1–20 — bevor deine Logik überhaupt läuft.

class RagQuery(BaseModel):
    question: str = Field(min_length=1, max_length=8_000)
    top_k: int = Field(default=4, ge=1, le=20)
Warum?

„Parse, don't validate": Sobald ein Objekt existiert, ist es garantiert gültig — der Rest des Codes braucht keine if-Prüfungen mehr. FastAPI generiert daraus außerdem automatisch die Swagger-Doku und die 422-Fehlerantworten (siehe Test test_chat_validation_rejects_empty_message).

Pydantic v2Field-ConstraintsSchema = Doku

Der LLM-Client: ein Vertrag, zwei Welten

app/services/llm_service.py

Zuerst wurde der Vertrag definiert (Protocol): Was muss jeder LLM-Client können? Dann die echte Implementierung: OllamaClient mit async httpx.

class LLMClient(Protocol):          # der Vertrag
    async def chat(self, prompt: str, ...) -> str: ...
    async def embed(self, texts: list[str]) -> list[list[float]]: ...

class OllamaClient:                 # die echte Welt
    async def chat(self, prompt, system=None, model=None) -> str:
        response = await self._client.post(f"{base}/api/chat", json=payload)
Warum Protocol? Warum async?

Protocol = strukturelle Typisierung: Der FakeLLMClient in den Tests erfüllt den Vertrag, ohne von irgendetwas zu erben. async, weil ein LLM-Aufruf sekundenlang dauert — währenddessen kann der Server andere Requests bedienen, statt blockiert zu warten. Das ist der Kern von „Starke Beherrschung in asynchroner Programmierung" aus der Stellenanzeige.

Die RAG-Bausteine: Parser, Chunker, Vector Store

app/rag/parser.py · chunker.py · vector_store.py

Drei kleine Module, jedes mit genau einer Aufgabe. Der Chunker hält Absätze zusammen und überlappt Chunks um 150 Zeichen. Der Vector Store misst Ähnlichkeit per Kosinus — bewusst in purem Python, ohne numpy.

def cosine_similarity(a, b) -> float:
    dot = sum(x * y for x, y in zip(a, b, strict=True))
    return dot / (norm_a * norm_b)   # 1.0 = gleiche Richtung = ähnliche Bedeutung
Warum Überlappung? Warum JSON statt Qdrant?

Überlappung, damit ein Satz an der Chunk-Grenze nicht zerrissen wird — er kommt in beiden Chunks vor und bleibt findbar. JSON zuerst, weil du so den Suchalgorithmus selbst verstehst, statt ihn hinter einer Datenbank zu verstecken. Die Schnittstelle (add_document, search) ist schon so geschnitten, dass Qdrant in Phase 3b als Drop-in daneben passt.

RagService: Komposition statt Monolith

app/rag/service.py

Der Service besitzt keine eigene „Intelligenz" — er ist der Dirigent, der die Bausteine in der richtigen Reihenfolge aufruft. Alle Abhängigkeiten kommen von außen herein.

class RagService:
    def __init__(self, llm: LLMClient, store: JsonVectorStore, ...):
        self._llm = llm        # injiziert — echt oder Fake

    async def ingest(self, filename, data, media_type) -> DocumentMeta:
        text   = parse_bytes(filename, data)      # 1 Parse
        pieces = chunk_text(text, ...)            # 2 Chunk
        vecs   = await self._llm.embed(pieces)    # 3 Embed
        self._store.add_document(meta, chunks)    # 4 Store
Warum?

Weil ingest() exakt der Pipeline aus deinem Konzept-Dokument entspricht (Upload → Parsing → Chunking → Embeddings → Vector DB) — nur als Code. Wer diese Methode liest, versteht das ganze System. Das nennt man „die Architektur schreit ihren Zweck heraus".

FastAPI-Verdrahtung: Lifespan, app.state, Depends

app/main.py · app/api/deps.py · app/api/routes.py

Beim Serverstart baut der lifespan-Kontext alle Objekte genau einmal und legt sie in app.state ab. Die Routen holen sie sich per Depends() — und beim Shutdown wird der HTTP-Client sauber geschlossen.

@asynccontextmanager
async def lifespan(app: FastAPI):
    llm = OllamaClient(settings.ollama_url, ...)
    app.state.rag = RagService(llm=llm, store=JsonVectorStore(...))
    yield                      # ← hier läuft der Server
    await llm.aclose()         # sauberer Shutdown

# in routes.py:
async def rag_query(body: RagQuery,
                    rag: Annotated[RagService, Depends(get_rag)]):
    return await rag.query(body.question, body.top_k)
Warum nicht einfach globale Variablen?

Drei Gründe: (1) Der Lebenszyklus ist explizit — erzeugt beim Start, geschlossen beim Stop, kein Zombie-Zustand. (2) Tests können die Objekte in app.state einfach austauschen. (3) Es gibt genau eine Stelle im Code, die entscheidet, welche konkreten Klassen laufen — die „Composition Root". Suchst du einen Bug in der Verkabelung, weißt du sofort, wo.

Tests ohne Netzwerk: der Fake-LLM-Trick

tests/conftest.py · test_api.py · test_chunker.py · test_vector_store.py

Der FakeLLMClient erfüllt das LLMClient-Protocol, braucht aber kein Ollama: Embeddings entstehen deterministisch aus Wort-Hashes — gleiche Wörter ergeben ähnliche Vektoren, also funktioniert sogar die semantische Suche im Test.

class FakeLLMClient:
    async def embed(self, texts): return [self._vector(t) for t in texts]

    @staticmethod
    def _vector(text):
        vector = [0.0] * 64          # Wort-Hash-Histogramm:
        for word in text.lower().split():   # gleiche Wörter →
            vector[hash_of(word) % 64] += 1.0 # ähnliche Vektoren
        return vector

Damit testet test_upload_and_rag_query den kompletten Fluss A + B: Upload → Chunking → „Embeddings" → Suche → Antwort. Und prüft, dass wirklich der richtige Chunk als Top-Quelle zurückkommt.

Warum keine echten LLM-Aufrufe im Test?

Echte LLM-Antworten sind langsam, kosten Ressourcen und sind nicht deterministisch — derselbe Test würde mal grün, mal rot. Die Tests hier laufen in 0,4 Sekunden, offline, in jeder CI. Das ist „Implementierung von Integrationstests für KI-Systeme" aus der Stellenanzeige — die Fähigkeit, die kaum ein Bewerber zeigt.

Selbst ausprobieren (10 Minuten)

Lernen heißt anfassen. Diese Sequenz führt einmal durch alle drei Datenflüsse — Ollama muss laufen.

  1. Starten: 
    cd backend
uv run uvicorn app.main:app --reload --port 8000
    Dann http://localhost:8000/docs öffnen — die Swagger-UI ist dein interaktiver Spielplatz.
  2. Fluss A — Dokument ingestieren: In Swagger POST /api/documents/upload aufklappen, eine PDF oder TXT hochladen. Danach in backend/data/vector_store.json hineinschauen: Du siehst deine Chunks samt Vektoren als Klartext. Das ist eine Vektordatenbank, entzaubert.
  3. Fluss B — Frage stellen: POST /api/rag/query mit einer Frage zum Dokument. Beachte die sources in der Antwort: Welcher Chunk hat welchen Score?
  4. Fluss C — Agenten: POST /api/agents/run mit {"message": "Baue eine Login-Seite"} — und lies, wie Planner, Developer und Reviewer aufeinander aufbauen.
  5. Experimentieren: Setze AIOS_CHUNK_SIZE=300, starte neu, lade dasselbe Dokument — wie verändern sich Trefferqualität und Scores? Genau solche Experimente sind „Bewertung von RAG-Systemen".
  6. Qualität prüfen wie in der CI: 
    uv run ruff check . && uv run mypy && uv run pytest

Qdrant

LangGraph

Evaluation & CI
